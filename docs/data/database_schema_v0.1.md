\# AI知识运营智能体平台数据库结构设计



> 文档版本：v0.1

> 文档状态：Draft

> 适用阶段：Phase 1｜数据层搭建

> 数据库：PostgreSQL 15.2

> 数据库名称：`ai\_knowledge\_agent`

> 应用角色：`ai\_kb\_app`

> 更新时间：2026-06-10



\---



\## 1. 文档目的



本文档用于冻结 Phase 1 第一版 PostgreSQL 数据结构，作为后续以下工作的唯一结构依据：



\* SQLAlchemy ORM 模型设计；

\* Pydantic Schema 设计；

\* Alembic 数据库迁移；

\* SKU Excel 导入；

\* 测试用例导入；

\* 对话日志存储；

\* Qdrant 元数据关联；

\* 自动化评测结果存储。



在本文档通过确认之前，不创建正式 Alembic 迁移文件。



\---



\## 2. Phase 1 数据库职责



PostgreSQL 负责保存以下结构化数据：



1\. SKU产品主数据；

2\. 会话与消息日志；

3\. 知识分片元数据；

4\. 测试用例；

5\. 评测运行记录；

6\. 单条评测结果；

7\. 数据导入批次和来源追踪。



PostgreSQL 不保存：



\* Embedding向量；

\* Qdrant索引文件；

\* 原始Excel和Markdown文件；

\* 模型权重；

\* 大型二进制附件；

\* 未脱敏的客户隐私数据。



Embedding向量存放于Qdrant，PostgreSQL仅保存对应的分片和关联标识。

\## 4. 通用设计规范



\### 4.1 表名与字段名



统一使用：



```text

snake\_case

小写英文字母

复数表名

```



示例：



```text

conversation\_sessions

evaluation\_results

created\_at

handoff\_required

```



禁止使用：



```text

驼峰命名

中文字段名

混合大小写

数据库保留字

```



\### 4.2 主键设计



所有业务表使用内部数值主键：



```sql

BIGINT GENERATED BY DEFAULT AS IDENTITY

```



业务编号使用独立唯一字段，例如：



```text

sku\_id

session\_id

chunk\_id

case\_id

run\_id

batch\_id

```



业务编号不能代替数据库内部主键。



\### 4.3 时间字段



所有时间字段使用：



```sql

TIMESTAMPTZ

```



数据库时区基线：



```text

Asia/Shanghai

```



所有核心表至少包含：



```text

created\_at

updated\_at

```



其中：



\* `created\_at` 在创建时写入；

\* `updated\_at` 由应用层在更新时维护；

\* v0.1阶段不创建数据库触发器。



\### 4.4 枚举字段



v0.1阶段不使用PostgreSQL原生ENUM。



统一采用：



```text

VARCHAR + CHECK约束

```



原因：



\* Alembic迁移更容易；

\* 后续增加状态值不需要修改数据库ENUM；

\* 更适合当前快速迭代阶段。



\### 4.5 多值字段



明确且稳定的字符串列表使用：



```sql

TEXT\[]

```



例如：



```text

secondary\_intents

expected\_sku\_ids

must\_contain\_all

must\_contain\_any

must\_not\_contain

allowed\_phrases

expected\_source

```



结构不固定或层级复杂的数据使用：



```sql

JSONB

```



例如：



```text

source\_references

config\_snapshot

summary\_metrics

raw\_result

error\_summary

```



\### 4.6 金额字段



Phase 1当前不创建正式产品价格表。



所有未核验金额：



```text

不得写入真实金额

不得使用占位金额

保持NULL

对应规则保持enabled=false

```

\# 5. 表结构设计



\## 5.1 `data\_import\_batches`



\### 5.1.1 表职责



记录每次数据文件导入，支持：



\* 文件版本追踪；

\* SHA256去重；

\* 导入数量统计；

\* 错误定位；

\* 数据来源审计；

\* 防止同一文件重复导入。



\### 5.1.2 字段设计



| 字段              | 类型           | 可空 | 约束与说明           |

| --------------- | ------------ | -: | --------------- |

| `id`            | BIGINT       |  否 | 内部主键，自增         |

| `batch\_id`      | UUID         |  否 | 业务批次ID，应用层生成，唯一 |

| `data\_type`     | VARCHAR(32)  |  否 | 数据类型            |

| `source\_file`   | VARCHAR(255) |  否 | 原始文件名           |

| `source\_path`   | VARCHAR(500) |  是 | 项目内相对路径         |

| `source\_sha256` | CHAR(64)     |  否 | 文件SHA256        |

| `record\_count`  | INTEGER      |  否 | 原始记录总数          |

| `success\_count` | INTEGER      |  否 | 成功记录数，默认0       |

| `failed\_count`  | INTEGER      |  否 | 失败记录数，默认0       |

| `status`        | VARCHAR(20)  |  否 | 导入状态            |

| `error\_summary` | JSONB        |  是 | 错误摘要            |

| `started\_at`    | TIMESTAMPTZ  |  是 | 导入开始时间          |

| `finished\_at`   | TIMESTAMPTZ  |  是 | 导入结束时间          |

| `created\_at`    | TIMESTAMPTZ  |  否 | 创建时间            |

| `updated\_at`    | TIMESTAMPTZ  |  否 | 更新时间            |



\### 5.1.3 `data\_type`允许值



```text

sku\_master

evaluation\_cases

spec\_questions

quality\_questions

price\_questions

logistics\_questions

business\_rules

```



\### 5.1.4 `status`允许值



```text

pending

running

success

partial\_success

failed

```



\### 5.1.5 约束



```text

record\_count >= 0

success\_count >= 0

failed\_count >= 0

success\_count + failed\_count <= record\_count

```



同一种数据类型下，相同SHA256文件不得重复成功导入。



建议唯一约束：



```text

UNIQUE(data\_type, source\_sha256)

```

\## 5.2 `products`



\### 5.2.1 表职责



保存由最新版 `sku\_master.xlsx` 导入的SKU产品主数据。



规格Handler必须查询该表，不允许从LLM记忆或Prompt中的静态清单获取产品参数。



\### 5.2.2 字段设计



| 字段                     | 类型           | 可空 | 约束与说明           |

| ---------------------- | ------------ | -: | --------------- |

| `id`                   | BIGINT       |  否 | 内部主键，自增         |

| `sku\_id`               | VARCHAR(32)  |  否 | SKU业务编号，唯一      |

| `product\_name`         | VARCHAR(255) |  否 | 产品名称            |

| `thread\_spec`          | VARCHAR(32)  |  否 | 完整螺纹规格，如M8×1.25 |

| `thread\_type`          | VARCHAR(8)   |  否 | 当前固定为M          |

| `thread\_diameter\_mm`   | NUMERIC(6,2) |  否 | 螺纹直径            |

| `thread\_pitch\_mm`      | NUMERIC(6,3) |  否 | 螺距              |

| `rod\_length\_mm`        | NUMERIC(8,2) |  否 | 杆长              |

| `ball\_diameter\_mm`     | NUMERIC(8,2) |  否 | 球径              |

| `taper\_ratio`          | VARCHAR(32)  |  是 | 锥度比，如1:10       |

| `material`             | VARCHAR(255) |  否 | 材质              |

| `surface\_treatment`    | VARCHAR(255) |  否 | 表面处理或工艺描述       |

| `oem\_reference\_number` | VARCHAR(128) |  否 | OEM对照号，必须按文本保存  |

| `min\_order\_qty`        | INTEGER      |  否 | 起订量             |

| `stock\_status`         | VARCHAR(32)  |  否 | 备货状态            |

| `lead\_time\_days`       | INTEGER      |  否 | 发货周期天数          |

| `is\_active`            | BOOLEAN      |  否 | 是否有效，默认true     |

| `import\_batch\_id`      | BIGINT       |  是 | 关联导入批次          |

| `source\_file`          | VARCHAR(255) |  否 | 来源文件            |

| `source\_row\_number`    | INTEGER      |  是 | Excel原始行号       |

| `created\_at`           | TIMESTAMPTZ  |  否 | 创建时间            |

| `updated\_at`           | TIMESTAMPTZ  |  否 | 更新时间            |



\### 5.2.3 数据转换规则



```text

锥度比“无” → NULL

空字符串 → NULL

OEM编号 → 文本

M8×1.25 → type=M，diameter=8，pitch=1.25

M10×1.5 → type=M，diameter=10，pitch=1.5

M12×1.25 → type=M，diameter=12，pitch=1.25

```



\### 5.2.4 约束



```text

sku\_id唯一

thread\_diameter\_mm > 0

thread\_pitch\_mm > 0

rod\_length\_mm > 0

ball\_diameter\_mm > 0

min\_order\_qty > 0

lead\_time\_days >= 0

source\_row\_number > 0 或为NULL

```



`taper\_ratio`允许：



```text

NULL

1:10

1:15

1:20

其他后续核验通过的1:N格式

```



\### 5.2.5 索引



必须建立：



```text

UNIQUE INDEX ON sku\_id

INDEX ON thread\_spec

INDEX ON thread\_diameter\_mm

INDEX ON thread\_pitch\_mm

INDEX ON material

INDEX ON stock\_status

INDEX ON oem\_reference\_number

INDEX ON is\_active

```



可增加联合索引：



```text

(thread\_diameter\_mm, thread\_pitch\_mm, is\_active)

```



用于螺纹规格筛选。

\## 5.4 `conversation\_messages`



\### 5.4.1 表职责



保存会话中的每一条用户、AI、人工或系统消息。



一条消息对应一行，禁止将整段对话堆叠在单个JSON字段中。



\### 5.4.2 字段设计



| 字段                   | 类型           | 可空 | 约束与说明                      |

| -------------------- | ------------ | -: | -------------------------- |

| `id`                 | BIGINT       |  否 | 内部主键                       |

| `message\_id`         | UUID         |  否 | 消息业务ID，唯一                  |

| `session\_id`         | BIGINT       |  否 | 关联conversation\_sessions.id |

| `role`               | VARCHAR(16)  |  否 | 消息角色                       |

| `content`            | TEXT         |  否 | 消息正文                       |

| `primary\_intent`     | VARCHAR(32)  |  是 | 主意图                        |

| `secondary\_intents`  | TEXT\[]       |  否 | 次级意图，默认空数组                 |

| `intent\_confidence`  | NUMERIC(5,4) |  是 | 意图置信度                      |

| `handler\_name`       | VARCHAR(64)  |  是 | 实际执行Handler                |

| `handoff\_required`   | BOOLEAN      |  否 | 是否转人工，默认false              |

| `handoff\_reason`     | VARCHAR(64)  |  是 | 转人工原因码                     |

| `source\_references`  | JSONB        |  否 | 数据来源，默认空数组                 |

| `processing\_time\_ms` | INTEGER      |  是 | 处理时延                       |

| `created\_at`         | TIMESTAMPTZ  |  否 | 消息时间                       |



\### 5.4.3 `role`允许值



```text

user

assistant

human

system

```



\### 5.4.4 `primary\_intent`允许值



```text

spec

quality

price

logistics

escalation

unknown

```



\### 5.4.5 约束



```text

content不得为空字符串

intent\_confidence必须在0到1之间

processing\_time\_ms >= 0

handoff\_required=false时，handoff\_reason可为空

handoff\_required=true时，handoff\_reason应填写

```



\### 5.4.6 外键策略



```text

conversation\_messages.session\_id

→ conversation\_sessions.id

ON DELETE CASCADE

```



删除测试会话时，可同步删除其消息。



正式生产环境是否允许删除会话，需要在Phase 2重新评估。



\### 5.4.7 索引



```text

UNIQUE INDEX ON message\_id

INDEX ON session\_id

INDEX ON primary\_intent

INDEX ON handoff\_required

INDEX ON created\_at

INDEX ON (session\_id, created\_at)

```

\## 5.5 `knowledge\_chunks`



\### 5.5.1 表职责



保存品质话术和物流FAQ的知识分片元数据。



本表不保存向量，只保存：



\* 原始文本；

\* 来源；

\* 分类；

\* Qdrant Point关联；

\* 审核状态；

\* 内容哈希。



\### 5.5.2 字段设计



| 字段                    | 类型           | 可空 | 约束与说明               |

| --------------------- | ------------ | -: | ------------------- |

| `id`                  | BIGINT       |  否 | 内部主键                |

| `chunk\_id`            | UUID         |  否 | 分片业务ID，唯一           |

| `knowledge\_type`      | VARCHAR(32)  |  否 | 知识类型                |

| `collection\_name`     | VARCHAR(64)  |  否 | Qdrant Collection名称 |

| `source\_file`         | VARCHAR(255) |  否 | 来源文件                |

| `source\_record\_id`    | VARCHAR(64)  |  是 | 来源问答ID              |

| `source\_version`      | VARCHAR(32)  |  是 | 来源版本                |

| `chunk\_text`          | TEXT         |  否 | 原始分片文本              |

| `content\_hash`        | CHAR(64)     |  否 | 分片内容SHA256          |

| `qdrant\_point\_id`     | UUID         |  是 | Qdrant Point ID     |

| `verification\_status` | VARCHAR(16)  |  否 | 审核状态                |

| `is\_active`           | BOOLEAN      |  否 | 是否启用，默认true         |

| `created\_at`          | TIMESTAMPTZ  |  否 | 创建时间                |

| `updated\_at`          | TIMESTAMPTZ  |  否 | 更新时间                |



\### 5.5.3 `knowledge\_type`允许值



```text

quality

logistics

```



\### 5.5.4 `verification\_status`允许值



```text

pending

verified

rejected

```



只有：



```text

verification\_status=verified

is\_active=true

```



的分片才能进入正式Qdrant Collection。



Phase 1开发测试期间可以使用单独的测试Collection处理pending数据，但不能与正式Collection混用。



\### 5.5.5 约束



```text

chunk\_text不得为空

content\_hash必须为64位SHA256

qdrant\_point\_id在写入Qdrant后必须填写

同一collection中的qdrant\_point\_id必须唯一

```



\### 5.5.6 索引



```text

UNIQUE INDEX ON chunk\_id

UNIQUE INDEX ON (collection\_name, qdrant\_point\_id)

INDEX ON knowledge\_type

INDEX ON verification\_status

INDEX ON is\_active

INDEX ON content\_hash

INDEX ON source\_record\_id

```

\### 5.7.5 索引

\## 5.8 `evaluation\_results`



\### 5.8.1 表职责



保存每次评测中每条测试用例的执行结果。



\### 5.8.2 字段设计



| 字段                    | 类型           | 可空 | 约束与说明                 |

| --------------------- | ------------ | -: | --------------------- |

| `id`                  | BIGINT       |  否 | 内部主键                  |

| `evaluation\_run\_id`   | BIGINT       |  否 | 关联evaluation\_runs.id  |

| `evaluation\_case\_id`  | BIGINT       |  否 | 关联evaluation\_cases.id |

| `actual\_intent`       | VARCHAR(16)  |  是 | 实际意图                  |

| `actual\_handoff`      | BOOLEAN      |  是 | 实际是否转人工               |

| `actual\_answer`       | TEXT         |  是 | 实际回答                  |

| `actual\_sku\_ids`      | TEXT\[]       |  否 | 实际SKU列表               |

| `handoff\_correct`     | BOOLEAN      |  是 | 转人工是否正确               |

| `sku\_correct`         | BOOLEAN      |  是 | SKU是否正确               |

| `contains\_all\_passed` | BOOLEAN      |  是 | 必含全部检查                |

| `contains\_any\_passed` | BOOLEAN      |  是 | 至少命中检查                |

| `excludes\_passed`     | BOOLEAN      |  是 | 禁止短语检查                |

| `human\_score`         | NUMERIC(3,1) |  是 | 人工评分                  |

| `raw\_result`          | JSONB        |  否 | 原始结果                  |

| `created\_at`          | TIMESTAMPTZ  |  否 | 创建时间                  |



\### 5.8.3 约束



```text

latency\_ms >= 0

human\_score在0到5之间

同一次run中同一case只能有一条结果

```



唯一约束：



```text

UNIQUE(evaluation\_run\_id, evaluation\_case\_id)

```



\### 5.8.4 外键策略



```text

evaluation\_results.evaluation\_run\_id

→ evaluation\_runs.id

ON DELETE CASCADE

```



```text

evaluation\_results.evaluation\_case\_id

→ evaluation\_cases.id

ON DELETE RESTRICT

```



删除评测运行时，可同步删除该次结果。



存在评测结果的测试用例不能直接删除，应通过：



```text

verification\_status=rejected

```



或版本管理停用。



\### 5.8.5 索引



```text

```

\# 7. 删除与停用策略



\## 7.1 产品数据



产品通常不物理删除，使用：



```text

is\_active=false

```



避免历史对话和评测结果失去关联依据。



\## 7.2 知识分片



知识分片更新时：



```text

旧分片 is\_active=false

新分片创建新chunk\_id

```



不覆盖旧分片内容，保证版本可追溯。



\## 7.3 测试用例



测试用例不直接删除。



不再使用时：



```text

verification\_status=rejected

```



\## 7.4 对话数据



Phase 1本地测试会话可以删除。



Phase 2接入真实平台后，需单独制定：



\* 数据保留周期；

\* 脱敏规则；

\* 客户数据删除规则；

\* 审计日志策略。

\# 8. 导入事务规则



每次文件导入必须采用单独事务。



标准流程：



```text

1\. 计算文件SHA256

2\. 创建data\_import\_batches记录

3\. 状态改为running

4\. 读取并校验数据

5\. 批量写入目标表

6\. 更新成功数和失败数

7\. 全部成功则提交事务

8\. 出错则回滚业务数据

9\. 导入批次状态改为failed或partial\_success

```



SKU主数据第一版采用：



```text

全量校验

全部通过后统一提交

任一关键记录错误则整体回滚

```



测试用例导入同样采用整体事务。

\# 9. 数据安全要求



数据库中不得保存：



\* 客户真实姓名；

\* 完整手机号；

\* 完整收货地址；

\* 身份证件信息；

\* 支付信息；

\* 未脱敏平台账号；

\* API Key；

\* 数据库密码；

\* LLM密钥。



Phase 2接入平台后，外部用户ID必须：



```text

脱敏

哈希化

或保存平台提供的不可逆标识

```



密钥只允许保存在：



```text

根目录 .env

```



不得写入数据库、日志或源码。
